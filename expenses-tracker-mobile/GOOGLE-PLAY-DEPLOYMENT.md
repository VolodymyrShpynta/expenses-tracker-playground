# Google Play Deployment Guide <!-- omit in toc -->

This guide walks you through publishing the **Spendium** mobile app to the
**Google Play Store**, end-to-end, from a fresh checkout to a production
release.

It assumes Windows + PowerShell as the dev environment (matching the rest of
this repo). [Step 3](#step-3--build-a-play-store-ready-aab) documents two
build paths — pure EAS cloud (zero local tooling) and `eas build --local`
(uses your local Android SDK with the same EAS-managed keystore). Both
feed into the same Step 4 upload and Step 5 `eas submit` automation, so
pick whichever matches your machine and skip the other.

> **Scope.** Android only — this repo doesn't currently target iOS in a
> production-ready way (the `ios.bundleIdentifier` is set in
> [app.json](app.json) but no Apple developer account workflow is wired up).
> For sideloading APKs to a phone for QA, see the
> [Building & Sideloading a Production APK](README.md#-building--sideloading-a-production-apk)
> section in the main README — this guide picks up where that leaves off
> and pushes to the Play Store.

## Prerequisites

You should already have the **EAS toolchain** working — `eas-cli` installed,
an Expo account, `npx eas login` done, the project linked via
`npx eas project:init`, and OAuth client IDs configured. All of that is the
same as for sideloading a `preview` build; the
[One-time setup](README.md#one-time-setup) section of the mobile README is
the canonical guide.

In addition, Play Store deployment requires:

| What                              | How / where                                | Notes                                                                                                                                       |
|-----------------------------------|--------------------------------------------|---------------------------------------------------------------------------------------------------------------------------------------------|
| **Google Play developer account** | https://play.google.com/console/signup     | **$25 one-time fee.** Identity verification (government ID + address) takes 1–3 days.                                                       |
| **Play App Signing**              | Enrolled on first upload (Step 4)          | Mandatory for new apps — Google holds the *app signing key*; you only manage the *upload key* (generated by EAS).                          |
| **Privacy policy URL (HTTPS)**    | Self-hosted page or a free generator — [GitHub Pages walkthrough in Step 8](#hosting-the-privacy-policy-and-a-landing-page-on-github-pages) | **Mandatory.** Even if "we collect nothing", you must declare so. Spendium stores data only on-device and in the user's own Drive/OneDrive. |
| **Store assets**                  | See [Step 6](#step-6--fill-out-the-play-console-listing) | Icon 512×512, feature graphic 1024×500, at least 2 phone screenshots. PNGs.                                                          |
| **Google Cloud service account**  | For automated `eas submit` — [Step 5](#step-5--wire-up-eas-submit-for-future-releases) | Lets EAS push subsequent `.aab` uploads to the Play Console without you opening the browser.                                  |

## Step 1 — Pre-flight: verify app.json and EAS config

The values below are what Play Console will see. Open
[app.json](app.json) and confirm:

| Field                            | Required value                                       | Current value (verify before release)                       |
|----------------------------------|------------------------------------------------------|-------------------------------------------------------------|
| `expo.name`                      | The user-visible app name shown on the device        | `"Spendium"`                                                |
| `expo.slug`                      | Stable internal slug (used by EAS)                   | `"expenses-tracker-mobile"` — fine, never user-visible      |
| `expo.version`                   | Semver shown in Play Store (e.g. `1.0.0`)            | **Bump from `0.0.1` before first release** — see note below |
| `expo.android.package`           | The Play Store application id, **immutable forever** | `"com.vshpynta.spendium"`                                   |
| `expo.android.adaptiveIcon`      | 1024×1024 PNG foreground + a solid background color  | Already set                                                 |
| `expo.android.permissions`       | Minimal set (currently just `INTERNET`)              | OK — sync uses the user's own cloud account, no extra perms |

> **Version vs. version code.** `expo.version` is the human-readable name
> ("1.0.0"). Google Play also requires an integer `versionCode` that must
> strictly increase with every upload. Because `eas.json` has
> `cli.appVersionSource: "remote"` and the `production` profile uses
> `autoIncrement: true`, EAS owns the `versionCode` for you — it
> auto-increments on every cloud build and stores the counter in your EAS
> account. **You don't need to edit `android.versionCode` in `app.json`.**

> **First-release version.** The Play Store rejects releases with version
> `0.x.x` as "alpha-looking" only in spirit, not by policy — but it's
> strongly recommended to ship `1.0.0` for the first production release.
> Bump [app.json](app.json) before the first `eas build --profile production`.

Then verify the EAS profile in [eas.json](eas.json):

```json
"production": {
  "channel": "production",
  "autoIncrement": true
}
```

(No `distribution: "internal"` here — that flag is what makes `preview`
emit an `.apk` for sideloading. Without it, `production` emits a Play-ready
`.aab`.)

## Step 2 — Create the app in Play Console

1. Go to https://play.google.com/console → **Create app**.
2. Fill in:
    - **App name:** `Spendium`
    - **Default language:** English (United States) — `en-US`
    - **App or game:** App
    - **Free or paid:** Free
    - **Declarations:** check both (Play policies + US export laws)
3. Click **Create app**.
4. Note your app's **package name** is now bound: it must match
   `expo.android.package` (`com.vshpynta.spendium`) on every upload. You
   cannot change either side later — a different package name = a different
   Play Store listing.

You'll now land on the app's **Dashboard**. The left rail has a "Set up
your app" checklist. We'll work through it in the next steps.

## Step 3 — Build a Play-Store-ready AAB

The mobile `README.md` already documents the **one-time EAS setup** (free
account, `npx eas login`, `npx eas project:init`, keystore generation on
first build, OAuth client IDs). If you haven't done it yet, follow the
[One-time setup](README.md#one-time-setup) section there first — Play
deployment reuses the exact same keystore and account as a `preview`
sideload build, so the setup is shared.

Two build paths produce a Play-ready `.aab`. They are mutually exclusive
— pick one and skip the other.

| Option   | Where the build runs | Toolchain you need               | Keystore managed by                         | Best for                                                                                  |
|----------|----------------------|----------------------------------|---------------------------------------------|-------------------------------------------------------------------------------------------|
| **A** — EAS Cloud | Expo's servers       | `eas-cli` only                  | Expo (auto-generated, downloadable)         | First release; CI; any machine without an Android SDK                                     |
| **B** — EAS Local | Your machine (macOS / Linux / WSL2 — **NOT Windows-native**) | `eas-cli` + Android SDK + JDK 17 | Expo (same keystore as Option A) | Faster iteration when you have Android Studio (or WSL2) installed; avoids EAS Cloud queue / limits |
| **C** — Pure Gradle | Your machine (Windows-native, macOS, Linux, WSL2 — all work) | Android SDK + JDK 17 + `keytool` | **You** (your own `.jks` file)              | Total offline pipeline; EAS Cloud unreachable or queued; EAS Local OOMs; you want full keystore control with no Expo account dependency |

All three produce the same artifact — a signed `.aab` — and feed into
the same [Step 4](#step-4--first-upload-manual-to-internal-testing)
upload. Options A & B can chain into [Step 5](#step-5--wire-up-eas-submit-for-future-releases)'s
`--auto-submit` (their build artifact has an EAS job id); Option C uploads
either through the Play Console UI or via `eas submit --path ./app-release.aab`
if you still have `eas-cli` installed. The EAS `production` profile differs
from `preview` (sideloading) only in two ways: output format (`.aab` vs
`.apk`) and `autoIncrement: true` on `versionCode`. Option C bypasses both
EAS flags entirely — see its section for the manual `versionCode`
workflow.

### Option A — EAS Cloud Build (recommended for first release)

Nothing to install locally beyond `eas-cli`. From
`expenses-tracker-mobile/`:

```powershell
npx eas whoami                                                # confirm you're logged in
npx eas build --platform android --profile production
```

**First-run prompts** (only on your very first `production` build for
this package):

1. *"Generate a new Android Keystore?"* → **Yes**. EAS creates the
   upload key, stores it server-side under your Expo account, and reuses
   it on every subsequent build (cloud or local).
2. *"What would you like your Android application id to be?"* — if it
   asks, accept the default (`com.vshpynta.spendium` from
   [app.json](app.json)). It should not ask if `expo.android.package` is
   already set.

**While it runs** the CLI prints a build-dashboard URL like
`https://expo.dev/accounts/<you>/projects/expenses-tracker-mobile/builds/<id>`.
Open it to watch logs in real time; the build is queued for ~30–60 s on
the free tier, then runs ~10–15 min.

**When it finishes:**

- The CLI prints a download URL
  (`https://expo.dev/artifacts/eas/<hash>.aab`). The same link is on
  the build dashboard under *Artifacts*.
- Download the `.aab` (browser, or
  `Invoke-WebRequest -Uri <url> -OutFile spendium-1.aab`). Save it
  somewhere you'll find again — you'll need it for the manual upload
  in [Step 4](#step-4--first-upload-manual-to-internal-testing).
- Verify the size is plausible (Spendium AABs land around 30–60 MB).
  Use `(Get-Item .\spendium-1.aab).Length / 1MB` to print it. A
  multi-hundred-MB or sub-1 MB file means something went wrong.
- Optional: `npx eas build:list --platform android --limit 1` re-prints
  the metadata (versionName, versionCode, status) if you lost the
  terminal output.

> **Free-tier limits.** Expo's free plan caps Android cloud builds at
> 30/month with no priority queue. If you hit the cap or queue is slow,
> switch to Option B for the rest of the month — same keystore, same
> versionCode counter, runs on your machine.

### Option B — EAS Local Build (`--local` flag)

Same command as Option A, `--local` flag added. EAS still owns the
keystore and the `versionCode` counter (so you must still be `eas login`'d
— it pulls credentials down on the first run), but the Gradle build
itself runs on your machine using the Android SDK + JDK 17 that Android
Studio installs.

> **Windows is NOT supported by `eas build --local`.** Running it on a
> Windows PowerShell session fails immediately with `Unsupported platform,
> macOS or Linux is required to build apps for Android`. The EAS local
> pipeline shells out to bash and assumes a POSIX environment. On Windows
> use one of:
>
> 1. **Stay on Option A (EAS Cloud).** Recommended for occasional releases
>    — zero extra setup, same artifact.
> 2. **WSL2 + Ubuntu** (the path documented below). One-time ~1-2 h setup,
>    then `--local` works exactly as on Linux.
> 3. **Option C — pure Gradle** in the mobile [README](README.md#option-c--pure-gradle-skip-eas-entirely),
>    which runs natively on Windows but bypasses EAS keystore + versionCode
>    management (you sign with your own `.jks` and bump `versionCode` by
>    hand).
>
> Native macOS and Linux can skip the WSL2 subsection and jump straight
> to *Prereqs*.

#### Windows: one-time WSL2 setup

Skip this subsection on macOS or Linux.

1. **Install WSL2 + Ubuntu** (PowerShell as admin, then reboot when it
   asks):

    ```powershell
    wsl --install -d Ubuntu
    ```

2. **Open Ubuntu** (Start menu → Ubuntu) and finish first-run user
   creation, then install the toolchain:

    ```bash
    sudo apt update
    sudo apt install -y openjdk-17-jdk unzip curl git
    # Node 20 LTS (matches EAS Cloud workers; nvm keeps things tidy)
    curl -o- https://raw.githubusercontent.com/nvm-sh/nvm/v0.39.7/install.sh | bash
    source ~/.bashrc
    nvm install 20 && nvm alias default 20
    ```

3. **Install the Android command-line tools** (Android Studio is *not*
   required inside WSL — only the SDK is):

    ```bash
    mkdir -p ~/Android/Sdk/cmdline-tools
    cd ~/Android/Sdk/cmdline-tools
    curl -L -o tools.zip https://dl.google.com/android/repository/commandlinetools-linux-11076708_latest.zip
    unzip tools.zip && mv cmdline-tools latest && rm tools.zip
    ```

4. **Set environment variables** (append to `~/.bashrc` so they persist):

    ```bash
    cat >> ~/.bashrc <<'EOF'
    export ANDROID_HOME=$HOME/Android/Sdk
    export PATH=$PATH:$ANDROID_HOME/cmdline-tools/latest/bin:$ANDROID_HOME/platform-tools
    export JAVA_HOME=/usr/lib/jvm/java-17-openjdk-amd64
    EOF
    source ~/.bashrc
    ```

5. **Install SDK platforms + build-tools and accept licenses**:

    ```bash
    sdkmanager "platform-tools" "platforms;android-35" "build-tools;35.0.0"
    yes | sdkmanager --licenses
    ```

6. **Reach the repo from inside WSL.** You have two choices — both are
   valid; pick based on how often you'll build.

   **Option 6a — Use your existing Windows checkout (simplest, recommended for occasional builds).**
   Your Windows clone at `C:\dev\tools\GitRepo\expenses-tracker-playground`
   is already visible from inside WSL as
   `/mnt/c/dev/tools/GitRepo/expenses-tracker-playground`. No second
   clone needed:

    ```bash
    cd /mnt/c/dev/tools/GitRepo/expenses-tracker-playground/expenses-tracker-mobile
    rm -rf node_modules package-lock.json    # only if you previously ran `npm install` from Windows — see callout below
    npm install
    npm install -g eas-cli
    eas login    # same Expo credentials as Windows
    ```

   **Option 6b — Clone a separate copy inside WSL's native filesystem (faster for frequent builds).**
   Gradle is ~3–5× faster on ext4 than on `/mnt/c/...` because
   cross-filesystem I/O over the 9P protocol is slow. Worth it if you'll
   build multiple times per day:

    ```bash
    mkdir -p ~/dev && cd ~/dev
    git clone <your-repo-url> expenses-tracker-playground
    cd expenses-tracker-playground/expenses-tracker-mobile
    npm install
    npm install -g eas-cli
    eas login    # same Expo credentials as Windows
    ```

   With this split, keep editing on Windows but build from `~/dev/...`
   inside WSL. Always commit and push from **one side only** to avoid
   Git index corruption from CRLF / permission-bit churn between
   Windows and WSL.

> **Always run `npm install` from Linux (WSL), never from Windows.** npm
> has a long-standing cross-platform bug ([npm/cli#4828](https://github.com/npm/cli/issues/4828))
> where `npm install` on Windows omits Linux-only optional native deps
> (e.g. `@emnapi/core`, `@emnapi/runtime`, several `@img/sharp-*`
> variants) from `package-lock.json`, even though the same deps are
> required on Linux. EAS Cloud workers run Linux + `npm ci`, which fails
> hard with `Missing: ... from lock file` when those entries are absent.
> The fix is to always regenerate the lockfile from a Linux shell —
> either WSL or the EAS Cloud workers themselves never see this drift
> because they only run Linux. If you previously ran `npm install` on
> Windows and committed the result, `rm -rf node_modules
> package-lock.json && npm install` from WSL rewrites it with all
> platform variants, and you should commit that lockfile in place of
> the Windows-only one.

7. **(WSL only) Cap Gradle / CMake parallelism in your Linux home.**
   `eas build --local` runs CMake to compile React Native's C++ for all
   four ABIs (`arm64-v8a`, `armeabi-v7a`, `x86`, `x86_64`). By default
   Ninja spawns one `clang++` per CPU core, and each peaks at
   ~500 MB – 1.5 GB resident. On a 16+ core WSL VM with the default
   ~15 GB RAM ceiling, the build is OOM-killed mid-link with **no error
   message** — you only see `Terminated` and `[ABORT] Received
   termination signal` in the log, because Linux `SIGKILL` does not
   reach the calling process's stderr.

   Capping parallelism takes **two** changes — one for Gradle, one for
   Ninja. They're separate knobs in separate places:

    ```bash
    # 1. Gradle daemon JVM size + max parallel Gradle tasks.
    #    These ARE settable in gradle.properties.
    mkdir -p ~/.gradle
    cat > ~/.gradle/gradle.properties <<'EOF'
    org.gradle.jvmargs=-Xmx4g -XX:MaxMetaspaceSize=1g -Dfile.encoding=UTF-8
    org.gradle.workers.max=2
    EOF

    # 2. Ninja (per-CMake-task) parallelism. This is read from the SHELL
    #    environment by the C++ build, NOT from gradle.properties — putting
    #    it in gradle.properties silently does nothing. Export it from
    #    ~/.bashrc so every WSL shell picks it up automatically.
    echo 'export CMAKE_BUILD_PARALLEL_LEVEL=4' >> ~/.bashrc
    source ~/.bashrc

    # Verify both
    cat ~/.gradle/gradle.properties
    echo "Ninja cap: $CMAKE_BUILD_PARALLEL_LEVEL"
    ```

   Peak memory math: `org.gradle.workers.max × CMAKE_BUILD_PARALLEL_LEVEL × ~1 GB`
   per concurrent compiler. The values above give `2 × 4 × 1 ≈ 8 GB`
   resident, comfortably under a 15 GB WSL ceiling. Bump them upward
   only if you've raised the WSL memory limit via `%UserProfile%\.wslconfig`
   and confirmed via `free -h` inside WSL.

   The `gradle.properties` file lives in your Linux home, **not** the
   project — Expo's prebuild wipes `android/gradle.properties` on every
   run, so a project-side tweak would be lost.

8. **(WSL only) Redirect `$TMPDIR` off `/tmp` onto the ext4 disk.**
   On WSL2 `/tmp` is a **tmpfs** mount sized at ~50% of the VM's RAM
   (~7.8 GB on a 15 GB VM — verify with `df -h /tmp`). EAS Local
   extracts the project tarball and runs the *entire* Gradle build under
   `/tmp/<user>/eas-build-local-nodejs/<uuid>/build/...`, so peak
   intermediates (per-ABI `.so` files in `merged_native_libs/` +
   `stripped_native_libs/` + Gradle's `fileHashes.bin` and
   `executionHistory.bin` + a copy of `node_modules`) easily exceed
   7.8 GB during `:app:mergeReleaseNativeLibs`. The failure surfaces
   late (~18 min in, after CMake) as:

    ```
    java.io.IOException: No space left on device
    ```

   even though `df -h /` shows hundreds of GB free on the real ext4
   disk — `/tmp` and `/` are different filesystems. Redirect EAS's
   scratch onto the ext4 root by exporting `TMPDIR` from `~/.bashrc`:

    ```bash
    mkdir -p /var/tmp/eas-build
    echo 'export TMPDIR=/var/tmp/eas-build' >> ~/.bashrc
    source ~/.bashrc
    # Verify it points at /dev/sdd (or whatever your WSL ext4 device is):
    df -h $TMPDIR | tail -1
    ```

   `/var/tmp` is on the same ext4 disk as `/home`, so EAS Local gets
   the full ~900 GB to spread across. The setting is harmless for
   non-EAS workloads.

#### Prereqs (all platforms)

Verify *before* running the build. On Windows these run inside the WSL2
Ubuntu shell; on macOS / Linux in your normal terminal:

```bash
java -version                              # 17.x recommended (matches EAS Cloud); 21 works but is not officially tested by EAS; 11 / 22+ will fail
echo $ANDROID_HOME                         # must print the SDK path
test -d $ANDROID_HOME/platforms/android-35       && echo OK   # must print OK
test -d $ANDROID_HOME/build-tools/35.0.0         && echo OK   # must print OK
eas whoami                                 # must print your Expo username, not 'Not logged in'
```

> **Why JDK 17 specifically?** EAS Cloud workers ship JDK 17, so 17 is
> the toolchain every cloud release is built against. JDK 21 is supported
> by AGP 8.5+ and usually works for local Spendium builds, but reproducing
> a cloud-identical artifact byte-for-byte (helpful when chasing
> production-only bugs) requires matching the cloud worker. JDK 11 fails
> outright (below AGP 8.x's minimum) and JDK 22+ is not yet officially
> supported by AGP.

If a SDK check fails on macOS / Linux with Android Studio installed,
open it → **More Actions → SDK Manager** → *SDK Platforms* tab and tick
**Android 15 (API 35)**; *SDK Tools* tab and tick **Android SDK
Build-Tools 35.0.0**. For JDK 17, point `JAVA_HOME` at Android Studio's
bundled JDK or install
[Eclipse Temurin 17](https://adoptium.net/temurin/releases/?version=17).

#### Run the build

From `expenses-tracker-mobile/` (inside WSL on Windows, or natively on
macOS / Linux):

```bash
eas build --platform android --profile production --local
```

**Artifact location.** The signed `.aab` is written to the working
directory as `expenses-tracker-mobile/build-<epoch>.aab` (e.g.
`build-1717123456789.aab`). The epoch suffix is unique per build, so you
can tell releases apart by date — no need to rename it.

**Timing.** The first `--local` build is slow (~20–30 min) because Gradle
downloads ~2 GB of dependencies, the Android NDK, and the React Native
build chain, and writes them into `~/.gradle`. Subsequent builds reuse
the cache and finish in 4–8 min. Free ≥10 GB of disk before the first
run (on WSL2 this is consumed inside the Ubuntu virtual disk, *not* on
your Windows `C:\` directly — `wsl --shutdown` followed by *Optimize-VHD*
can reclaim it later).

> **Keystore backup.** The first `eas build` for this `package` generates
> the **upload key** and stores it in your Expo account. For Play Store
> deployment the stakes are higher than for sideloading: lose the keystore
> and Google won't let you ship updates. Run
> `eas credentials -p android` → **Download keystore** to back up the
> `.jks` and its passwords in a password manager.

### Option C — Pure Gradle (no EAS, no cloud)

When to pick this:

- EAS Cloud is queued or down, EAS Local OOMs or fails (see the
  [troubleshooting table](#troubleshooting)), and you need a Play-ready
  AAB *now*.
- You want a fully offline, EAS-free pipeline and are happy to manage
  your own keystore + `versionCode` instead of delegating those to Expo.
- You're on **Windows-native** (no WSL) — plain `gradlew.bat` builds work
  here even though `eas build --local` does not.

Trade-offs vs Options A/B:

| Concern              | Options A & B (EAS)                           | Option C (Pure Gradle)                                                        |
|----------------------|-----------------------------------------------|-------------------------------------------------------------------------------|
| Keystore             | Expo-managed, downloadable from your account  | **You** own the `.jks` — lose it and you can never update the app on Play     |
| `versionCode`        | `autoIncrement: true` in `eas.json`           | You bump `expo.android.versionCode` in [app.json](app.json) by hand           |
| First-upload submit  | Manual via Play Console UI (Step 4)           | Manual via Play Console UI (Step 4) — same                                    |
| Steady-state submit  | `eas submit --auto-submit` chained to build   | `eas submit --path ./app-release.aab` (needs `eas-cli`), or upload by browser |
| Native toolchain     | Hidden by EAS (Cloud) or set up for `--local` | Always required: Android SDK + JDK 17 — same as the README's dev-client      |

**Host prerequisites** — identical to the dev-client section in the README:
JDK 17–21 with `JAVA_HOME` set, Android SDK platform 34+ with NDK 27.x and
CMake 3.22+, and (on Windows) long-path support enabled. Full instructions
in [README → Building a Local Dev Client](README.md#-building-a-local-dev-client-npx-expo-runandroid).

#### One-time setup

**1. Generate your upload keystore** — this `.jks` file and its passwords
are what Play will use to verify every future update. Pick passwords from a
password manager.

```powershell
# Windows PowerShell (Linux / macOS / WSL: same command, no .exe and use $HOME)
mkdir $env:USERPROFILE\keystores -Force
keytool -genkey -v `
  -keystore $env:USERPROFILE\keystores\spendium-upload.jks `
  -alias spendium-upload `
  -keyalg RSA -keysize 2048 -validity 9125 `
  -dname "CN=Spendium, O=Spendium, C=US"
```

Back up `spendium-upload.jks` to **at least two** places (password manager
attachment, encrypted external drive, etc.). Without it Play Console will
refuse every future upload signed with a different key.

**2. Expose the keystore credentials as environment variables.** Add these
to `$PROFILE` (PowerShell) or `~/.bashrc` (Linux / WSL) so every shell
picks them up:

```powershell
# PowerShell ($PROFILE)
$env:SPENDIUM_KEYSTORE_PATH     = "$env:USERPROFILE\keystores\spendium-upload.jks"
$env:SPENDIUM_KEYSTORE_PASSWORD = "<store password from step 1>"
$env:SPENDIUM_KEY_ALIAS         = "spendium-upload"
$env:SPENDIUM_KEY_PASSWORD      = "<key password from step 1>"
```

#### Building the AAB

**1. Bump versions in [app.json](app.json):**

- `expo.version` — human-readable, e.g. `"1.0.0"` → `"1.1.0"`.
- `expo.android.versionCode` — integer, **strictly greater** than the
  last one Play has seen. If you previously built with EAS, query
  `eas build:version:get -p android` and start above that.

**2. Regenerate the native project:**

```powershell
cd expenses-tracker-mobile
npx expo prebuild --platform android --clean
```

`--clean` discards the previous `android/` (which is `.gitignore`d) so the
output is reproducible from `app.json`.

**3. Patch the generated `android/app/build.gradle` to use a release
signing config.** Expo's prebuild template signs `release` with the
*debug* key by default — Play Console rejects that. Edit
`android/app/build.gradle` to look like this:

```diff
   signingConfigs {
     debug {
       storeFile file('debug.keystore')
       storePassword 'android'
       keyAlias 'androiddebugkey'
       keyPassword 'android'
     }
+    release {
+      storeFile file(System.getenv("SPENDIUM_KEYSTORE_PATH"))
+      storePassword System.getenv("SPENDIUM_KEYSTORE_PASSWORD")
+      keyAlias System.getenv("SPENDIUM_KEY_ALIAS")
+      keyPassword System.getenv("SPENDIUM_KEY_PASSWORD")
+    }
   }
   buildTypes {
     release {
-      signingConfig signingConfigs.debug
+      signingConfig signingConfigs.release
       // ...
     }
   }
```

Automation one-liner (skip if you prefer editing by hand):

```bash
# Linux / WSL / macOS
sed -i.bak \
  -e 's|signingConfig signingConfigs.debug|signingConfig signingConfigs.release|' \
  -e '/signingConfigs {/a\        release {\n            storeFile file(System.getenv("SPENDIUM_KEYSTORE_PATH"))\n            storePassword System.getenv("SPENDIUM_KEYSTORE_PASSWORD")\n            keyAlias System.getenv("SPENDIUM_KEY_ALIAS")\n            keyPassword System.getenv("SPENDIUM_KEY_PASSWORD")\n        }' \
  android/app/build.gradle
```

Because `android/` is regenerated on every `prebuild`, this patch must be
reapplied each release — or, for a permanent fix, factor it into a small
Expo config plugin (`@expo/config-plugins` → `withAppBuildGradle`) checked
into the repo.

**4. Build the bundle:**

```powershell
cd android
.\gradlew.bat bundleRelease    # Windows
# ./gradlew bundleRelease      # Linux / macOS / WSL
```

**5. Output:**

```
expenses-tracker-mobile/android/app/build/outputs/bundle/release/app-release.aab
```

**6. Verify the AAB is signed with *your* upload key** (not the debug key)
before uploading, otherwise Play Console will reject it:

```powershell
# Pull the signing cert fingerprint out of the .aab and compare to your keystore.
keytool -printcert -jarfile app\build\outputs\bundle\release\app-release.aab | Select-String SHA1
keytool -list -v -keystore $env:USERPROFILE\keystores\spendium-upload.jks -alias spendium-upload | Select-String SHA1
```

The two `SHA1:` lines must match. If they don't, you forgot the patch in
step 3 (or your environment variables aren't set in this shell).

**7. Upload manually** via Play Console (Step 4 — same as Options A/B for
the first upload). For steady-state, see Step 10.

> **Don't `git add android/`.** It's intentionally `.gitignore`d —
> committing it freezes the prebuild output against later Expo SDK
> upgrades and makes `npx expo prebuild --clean` a no-op. If you need to
> persist *any* native patches, do it through an Expo config plugin, not
> by checking in the generated folder.

## Step 5 — Wire up `eas submit` for future releases

Once the first upload is on Internal testing, automate subsequent
uploads. This needs a Google Cloud **service account** with Play Console
permissions.

1. **Create the service account** (one-time):
    - In Google Cloud Console, create a project (or use an existing one

    - Enable the **Google Play Android Developer API**.

3. **Wire it into `eas.json`**:

    ```json

      "production": {
        "android": {
          "serviceAccountKeyPath": "./play-store-service-account.json",

        }
      }
    }
    ```

#### Branding verification on a `*.github.io` subdomain (ownership trap)

When you click **Verify branding** in *Google Auth Platform → Branding*,
Google opens a *Branding verification issues* dialog complaining:

> The website of your home page URL `https://<user>.github.io/spendium-site/` is not registered to you. Verify ownership of your home page.

The fix is **not** in Cloud Console — it's in **Google Search Console**,
signed in as the same Google account that owns this Cloud project.
Cloud Console calls into Search Console to confirm "this account owns
this URL"; merely hosting a verification file does nothing on its own
until you click *Verify* inside Search Console.

1. **Open [Google Search Console](https://search.google.com/search-console)
   under the Cloud-project owner.** Check the avatar top-right — verifying
   under a different Google account is the #1 cause of repeat failures.
   If you have multiple Google accounts signed in, use an incognito window
   with only the Cloud-project owner signed in to avoid Google silently
   using the wrong session.

2. **Add the property as *URL prefix*, NOT *Domain*.** DNS verification
   is impossible on `*.github.io` because you don't control github.io's
   DNS. Paste the full home-page URL **with trailing slash**:
   `https://<user>.github.io/spendium-site/`. Pick the **HTML file**
   verification method.

3. **Drop the file at the repo root and push.** Move the offered
   `google<hash>.html` into `spendium-site/`, commit, and push to `main`.
   Wait ~30 s for the Pages rebuild, then in a browser confirm
   `https://<user>.github.io/spendium-site/google<hash>.html` returns the
   one-line body `google-site-verification: google<hash>.html`.

4. **Click *Verify* in Search Console.** Should turn green within seconds
   ("Ownership verified automatically"). If it doesn't, you skipped
   step 3 (the file isn't reachable yet) or you're signed in as the
   wrong Google account — the filename it offers you is deterministic
   per `(account, property)`, so a different filename than yours is
   proof of a wrong-account session.

5. **Wait 5 minutes.** Cloud Console caches Search Console membership
   and doesn't always re-query immediately. Re-clicking too fast also
   counts against a daily re-verification quota.

6. **Back in Cloud Console → *Branding* → *View issues*** → tick
   **"I have fixed the issues"** → *Proceed*. The yellow banner
   disappears within a minute. **Do not** pick *"I believe the issues
   found are incorrect"* — that opens a slow manual-review ticket
   (days to weeks) and is meant only as a last resort.

Two non-obvious gotchas:

- **The verification HTML file must stay in the repo forever.** Google
  re-checks it periodically; if it disappears, your verified status
  revokes and the OAuth consent screen falls back to "unverified app".
  Commit it once and leave it alone — don't delete it during cleanup.
- **Search Console *indexing* status is irrelevant.** After verifying,
  the Search Console overview will show "Data is processing — try again
  in one day" for *Performance* and *Indexing*. That's about web search
  crawling, completely unrelated to OAuth branding. Close the tab and
  ignore it.

If the automated re-check still fails after a successful Search Console
verification + 5-minute wait, you've hit a real Cloud Console quirk
with `*.github.io` subdomains where its "registered to you" check is
stricter than Search Console's. Two pragmatic options:

- **Skip branding verification entirely.** It is *optional*. Without it,
  the OAuth consent screen shows your developer email instead of the
  app name + logo. Drive sign-in still works, the app still works —
  users just see a slightly less polished consent screen. For a hobby /
  side-project app this is the pragmatic choice; click *Cancel* in the
  dialog and accept the yellow banner.
- **Move to a real custom domain.** Buy a cheap domain (~$10/year on
  Cloudflare / Porkbun / Namecheap), follow the
  [custom-domain upgrade path below](#when-you-outgrow-it-custom-domain),
  then in Search Console re-verify with the **Domain** option
  (DNS TXT record — always works, sidesteps the `*.github.io` quirk).

#### When you outgrow it (custom domain)

When you're ready for `https://spendium.app/privacy` instead of the
`github.io` subpath, register the domain, add a `CNAME` file to
`spendium-site/` containing just the domain, and set the DNS records per
GitHub's [custom-domain instructions](https://docs.github.com/pages/configuring-a-custom-domain-for-your-github-pages-site).
Then update the URLs above. Play Console accepts a privacy-policy URL
change any time without re-review — just paste the new URL and save.

## Step 9 — Promote Internal → Closed → Open → Production

Google strongly recommends staging through testing tracks before going
public. Each track uses the same `.aab`; you only choose **which track**
the release lands on.

| Track                | Audience                                     | When to use                                                                                                                       |
|----------------------|----------------------------------------------|-----------------------------------------------------------------------------------------------------------------------------------|
| **Internal testing** | Up to 100 testers you list by email          | Smoke test the build on real devices via the Play Store install flow (catches things that local sideloading misses, e.g. App Signing). |
| **Closed testing**   | Larger tester groups (e.g. 100s of users)    | Required for first-time publishers: Google now demands ~20 testers run a Closed track for ~14 days before allowing Production.    |
| **Open testing**     | Anyone with the opt-in URL                   | Optional public beta.                                                                                                              |
| **Production**       | Everyone in the targeted countries           | Final release.                                                                                                                     |

Promotion is one click: **Internal testing → Promote release → Production**
(or any intermediate track). The `.aab` is reused; only the track changes.

> **First-app reality check.** As of late 2024, new Google Play developer
> accounts are required to run a **Closed testing track with at least 20
> opted-in testers for 14 days** before being eligible to publish to
> Production. Plan for this gating — share the opt-in URL with at least
> 20 people the day you create the Closed track.

## Step 10 — Shipping subsequent updates

```powershell
# 1. Bump the human-readable version in app.json (e.g. "1.0.0" -> "1.1.0").
#    Options A/B: versionCode auto-increments via EAS — don't touch it.
#    Option C:    bump expo.android.versionCode in app.json by hand.

# 2. Build + submit. Pick the path you used in Step 3:

# Option A (EAS Cloud):
npx eas build --platform android --profile production --auto-submit

# Option B (EAS Local):
npx eas build --platform android --profile production --local
npx eas submit --platform android --path ./build-*.aab

# Option C (Pure Gradle):
npx expo prebuild --platform android --clean
# Reapply the release-signingConfig patch from Step 3 → Option C step 3
#   (or use the sed one-liner / your config plugin).
cd android ; .\gradlew.bat bundleRelease ; cd ..
# Either upload android/app/build/outputs/bundle/release/app-release.aab
# via the Play Console UI, or, if you still have eas-cli:
npx eas submit --platform android --path android/app/build/outputs/bundle/release/app-release.aab

# 3. In Play Console -> Internal testing, write release notes (per locale)
#    and start rollout to Internal testing.

# 4. Smoke-test on a tester device. When happy:
#    Internal testing -> Promote release -> Production -> Start staged rollout.
```

Staged rollout starts at 1–10 % of users and you bump it up over days.
If a crash spike shows in Play Console → Quality → Android vitals,
**Halt rollout** stops new installs immediately (existing installs keep
the bad build until you ship a fix).

## Optional — EAS Update (OTA JS-only patches)

For JavaScript-only fixes (a typo, a logic bug in `src/`, no native
changes), you can push an over-the-air update without going through the
Play Store review:

```powershell
npx eas update --channel production --message "fix: typo in settings"
```

This requires `expo-updates` to be installed and configured. **Native
changes** (new dependency that ships native code, `app.json` plugin
tweaks, icon updates, permission changes) still require a fresh
`eas build` + Play Store release — EAS Update can only swap the JS
bundle.

Configure once via:

```powershell
npx expo install expo-updates
npx eas update:configure
```

…then add `runtimeVersion` to [app.json](app.json) so a JS bundle is only
served to compatible native builds.

## Troubleshooting

| Symptom                                                              | Cause                                                                                          | Fix                                                                                                                                                                                                                                                                       |
|----------------------------------------------------------------------|------------------------------------------------------------------------------------------------|-----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| `Version code X has already been used`                               | EAS's auto-increment got out of sync with what Play already has (e.g. you uploaded manually).  | Run `npx eas build:version:set` (interactive) to align the EAS counter with the highest value Play has seen. The flag form `--platform android --value 42` skips the prompts. Alternatively, switch `cli.appVersionSource` to `"local"` in [eas.json](eas.json) and set `expo.android.versionCode` in [app.json](app.json) directly — then EAS reads from there instead of its remote counter. |
| `Your app is using a non-compliant version of the Play Core library` | A transitive RN dep still imports the legacy Play Core API.                                    | Bump to Expo SDK 55+ (already on it). If the error persists, run `npx expo-doctor` and follow its upgrade hints.                                                                                                                                                       |
| `App not installed: package conflicts with an existing package`      | A sideloaded `preview` APK on the test phone uses the same package id but a different signing key. | Uninstall the sideloaded copy first: `adb uninstall com.vshpynta.spendium`. Play-installed and dev-signed builds can't coexist on one device.                                                                                                                          |
| Play Console: "Privacy policy URL is required"                       | The URL field is empty or returns non-200 to Google's crawler.                                 | Make sure the URL is HTTPS and publicly reachable (no Cloudflare bot-blocker, no auth wall, no robots-disallow).                                                                                                                                                       |
| Play Console: "Your app cannot be promoted to production"            | First-publisher 20-tester / 14-day Closed-track requirement not met.                           | Open Closed testing → ensure ≥ 20 opted-in testers → wait the full 14 days. There is no opt-out.                                                                                                                                                                       |
| `eas submit` fails with `Permission denied`                          | The service account doesn't have Play Console access yet, or its permissions weren't saved.    | Play Console → Users and permissions → confirm the service-account email is listed and "App access" includes your app. Permission changes can take a few minutes to propagate.                                                                                         |
| `eas build --local` exits with `Unsupported platform, macOS or Linux is required to build apps for Android` | You're on Windows PowerShell. EAS's local pipeline requires a POSIX shell. | Use **Option A** (cloud), **Option C** (pure Gradle — [see Step 3](#option-c--pure-gradle-no-eas-no-cloud)), or set up **WSL2 + Ubuntu** and run from inside WSL (see [Option B → Windows: one-time WSL2 setup](#windows-one-time-wsl2-setup)). |
| EAS Cloud build fails with `npm ci ... can only install packages when your package.json and package-lock.json ... are in sync` (e.g. `Missing: @emnapi/core@... from lock file`) | Your lockfile drifted from `package.json`. Common trigger: you accepted the EAS prompt to auto-install a missing package (`expo-updates`, `expo-doctor`, etc.) — the install updates `package.json` but on some npm versions / OSes leaves optional/peer transitive deps out of `package-lock.json`. EAS Cloud uses `npm ci`, which refuses to run with any drift. | Locally: `npm install` (regenerates the lockfile), then `git add package.json package-lock.json && git commit`, then re-run `eas build`. Never edit `package-lock.json` by hand and never run `npm audit fix --force`. |
| Same error as the row above, but `npm install` on Windows didn't fix it — EAS Cloud still reports `Missing: @emnapi/core@...` (or other Linux-only deps like `@img/sharp-linux-*`) after you committed the regenerated lockfile. | npm's cross-platform `optionalDependencies` bug ([npm/cli#4828](https://github.com/npm/cli/issues/4828)) — `npm install` on Windows skips and *omits from the lockfile* native deps that don't apply to Windows, even though Linux builds need them. EAS Cloud runs Linux + `npm ci`, which then fails. | Regenerate the lockfile from a Linux shell (WSL or any Linux/macOS box): `rm -rf node_modules package-lock.json && npm install`, then commit. Going forward, always run `npm install` from WSL on this repo — see [Option B → step 6](#windows-one-time-wsl2-setup) callout. |
| OAuth (Google Drive / OneDrive) breaks only in the Play-Store build  | Play App Signing strips the upload key on Google's servers and re-signs the APK with Google's own **app signing key** before delivery, so the SHA-1 attested at OAuth runtime is the *app signing* SHA-1 — not the upload key SHA-1 you originally registered. The redirect handler `com.vshpynta.spendium:/oauth2redirect` is then signed by a key Google doesn't recognise on your OAuth client. | In Google Cloud Console → **Google Auth Platform → Clients** → your Android OAuth client, **replace** the existing SHA-1 with the **app signing key** SHA-1. Get it from Play Console: **Protected with Play** → expand **Play Store protection** → **Protect app signing key** → **Manage Play app signing** button (direct URL: `https://play.google.com/console/u/0/developers/<DEVELOPER_ID>/app/<APP_ID>/keymanagement`). Copy from the **App signing key certificate** block (top one — *not* **Upload key certificate** below). The current Google UI only accepts **one** SHA-1 per Android client (the *Add fingerprint* button no longer exists), so swap rather than try to add. Trade-off: sideloaded local-debug builds can no longer sign in to Drive until the SHA-1 is swapped back — acceptable for now since Play-installed builds are what real users run. |
| Branding dialog: *"The website of your home page URL '...github.io/...' is not registered to you"* | Google Auth Platform's *Verify branding* flow calls into Google Search Console under the same account for proof of ownership of the home-page URL. Hosting a `google<hash>.html` file at the URL alone does **not** count — you must explicitly click *Verify* inside Search Console, signed in as the Cloud-project owner, against a `URL prefix` property (not `Domain`, which requires DNS access you don't have on `*.github.io`). | Full walk-through with the `*.github.io` ownership-trap gotchas: [Step 8 → Branding verification on a `*.github.io` subdomain](#branding-verification-on-a-githubio-subdomain-ownership-trap). TL;DR: verify in [Search Console](https://search.google.com/search-console) with **URL prefix** + **HTML file**, wait 5 min, then in Cloud Console pick **"I have fixed the issues"** (never *"I believe the issues found are incorrect"* — that opens a slow manual-review ticket). If the automated re-check still fails after a confirmed green Search Console verification, branding is optional — cancel the dialog and ship without it, or switch to a real custom domain. |
| Gradle exits immediately with `Timeout waiting to lock journal cache (~/.gradle/caches/journal-1). It is currently in use by another process. Owner PID: …` | A previous `eas build` was interrupted (Ctrl+C, OOM, terminal crash) and left a Gradle daemon JVM alive holding the journal lock. Gradle daemons idle for 3 hours by default, so they don't release on their own. | `pkill -f GradleDaemon ; rm -f ~/.gradle/caches/journal-1/journal-1.lock` from inside WSL, then retry. If it happens often, add `org.gradle.daemon=false` to `~/.gradle/gradle.properties` — slower startup per build but no orphan daemons. |
| `eas build --local` exits with `Terminated` + `[ABORT] Received termination signal` mid-CMake, **no error message** above the abort line. | OOM killer. CMake compiles RN's C++ for 4 ABIs in parallel; on a many-core WSL VM (default = one `clang++` per core, ~22 jobs) total resident memory exceeds the VM's RAM limit and the kernel kills the build with `SIGKILL`, which doesn't print anything. | Cap **both** Gradle workers (`org.gradle.workers.max` in `~/.gradle/gradle.properties`) **and** Ninja parallelism (`export CMAKE_BUILD_PARALLEL_LEVEL=4` in `~/.bashrc`) — see [Option B → step 7](#windows-one-time-wsl2-setup). Note: `CMAKE_BUILD_PARALLEL_LEVEL` is *not* a Gradle property and putting it in `gradle.properties` silently does nothing — it must be a real shell env var. As a bigger hammer, also raise the WSL memory limit in `%UserProfile%\.wslconfig` (`memory=12GB`, `swap=8GB`, then `wsl --shutdown`). |
| `eas build --local` fails late (~18 min in, past CMake) at `:app:mergeReleaseNativeLibs` with `java.io.IOException: No space left on device`, even though `df -h /` inside WSL shows hundreds of GB free. | WSL2's `/tmp` is a **tmpfs** sized at ~50% of the VM's RAM (~7.8 GB), separate from the ext4 root. EAS Local extracts the project and runs the entire Gradle build under `/tmp/<user>/eas-build-local-nodejs/<uuid>/...`, and peak intermediates (per-ABI `.so` copies in `merged_native_libs/` + `stripped_native_libs/` + Gradle cache + a copy of `node_modules`) blow past tmpfs. Confirm with `findmnt /tmp` showing `tmpfs`. | Redirect EAS scratch onto the ext4 disk: `mkdir -p /var/tmp/eas-build && echo 'export TMPDIR=/var/tmp/eas-build' >> ~/.bashrc && source ~/.bashrc`. See [Option B → step 8](#windows-one-time-wsl2-setup) for the full explanation. Resizing tmpfs (`sudo mount -o remount,size=32G /tmp`) also works but permanently reserves that much VM RAM. |
| WSL session won't start: `A fatal error was returned by plugin 'DefenderforEndpointPlug-in'. Error code: Wsl/Service/CreateInstance/Plugin/0x8007274c` | Microsoft Defender for Endpoint's WSL integration timed out reaching its management endpoint (`*.endpoint.security.microsoft.com`). Often hits after a VPN reconnect, an EDR service restart, or a hibernation. Not project-related. | From Windows PowerShell: `wsl --shutdown ; wsl -d Ubuntu`. ~80% of cases this is a transient hiccup and the restart clears it. If it persists, check `Get-Service Sense, WdNisSvc, WinDefend` (all should be `Running`); if `Sense` is degraded, `Restart-Service Sense -Force` (admin) and retry. The WSL plug-in lives outside your build environment — your project files, `~/.gradle/gradle.properties`, etc. survive the restart unchanged. |
