# Google Play Deployment Guide <!-- omit in toc -->

This guide walks you through publishing the **Spendium** mobile app to the
**Google Play Store**, end-to-end, from a fresh checkout to a production
release.

It assumes Windows + PowerShell as the dev environment (matching the rest of
this repo). [Step 3](#step-3--build-a-play-store-ready-aab) documents two
build paths — pure EAS cloud (zero local tooling) and `eas build --local`
(uses your local Android SDK with the same EAS-managed keystore). Both
feed into the same Step 4 upload and Step 5 `eas submit` automation, so
pick whichever matches your machine and skip the other.

> **Scope.** Android only — this repo doesn't currently target iOS in a
> production-ready way (the `ios.bundleIdentifier` is set in
> [app.json](app.json) but no Apple developer account workflow is wired up).
> For sideloading APKs to a phone for QA, see the
> [Building & Sideloading a Production APK](README.md#-building--sideloading-a-production-apk)
> section in the main README — this guide picks up where that leaves off
> and pushes to the Play Store.

## Prerequisites

You should already have the **EAS toolchain** working — `eas-cli` installed,
an Expo account, `npx eas login` done, the project linked via
`npx eas project:init`, and OAuth client IDs configured. All of that is the
same as for sideloading a `preview` build; the
[One-time setup](README.md#one-time-setup) section of the mobile README is
the canonical guide.

In addition, Play Store deployment requires:

| What                              | How / where                                | Notes                                                                                                                                       |
|-----------------------------------|--------------------------------------------|---------------------------------------------------------------------------------------------------------------------------------------------|
| **Google Play developer account** | https://play.google.com/console/signup     | **$25 one-time fee.** Identity verification (government ID + address) takes 1–3 days.                                                       |
| **Play App Signing**              | Enrolled on first upload (Step 4)          | Mandatory for new apps — Google holds the *app signing key*; you only manage the *upload key* (generated by EAS).                          |
| **Privacy policy URL (HTTPS)**    | Self-hosted page or a free generator       | **Mandatory.** Even if "we collect nothing", you must declare so. Spendium stores data only on-device and in the user's own Drive/OneDrive. |
| **Store assets**                  | See [Step 6](#step-6--fill-out-the-play-console-listing) | Icon 512×512, feature graphic 1024×500, at least 2 phone screenshots. PNGs.                                                          |
| **Google Cloud service account**  | For automated `eas submit` — [Step 5](#step-5--wire-up-eas-submit-for-future-releases) | Lets EAS push subsequent `.aab` uploads to the Play Console without you opening the browser.                                  |

## Step 1 — Pre-flight: verify app.json and EAS config

The values below are what Play Console will see. Open
[app.json](app.json) and confirm:

| Field                            | Required value                                       | Current value (verify before release)                       |
|----------------------------------|------------------------------------------------------|-------------------------------------------------------------|
| `expo.name`                      | The user-visible app name shown on the device        | `"Spendium"`                                                |
| `expo.slug`                      | Stable internal slug (used by EAS)                   | `"expenses-tracker-mobile"` — fine, never user-visible      |
| `expo.version`                   | Semver shown in Play Store (e.g. `1.0.0`)            | **Bump from `0.0.1` before first release** — see note below |
| `expo.android.package`           | The Play Store application id, **immutable forever** | `"com.vshpynta.spendium"`                                   |
| `expo.android.adaptiveIcon`      | 1024×1024 PNG foreground + a solid background color  | Already set                                                 |
| `expo.android.permissions`       | Minimal set (currently just `INTERNET`)              | OK — sync uses the user's own cloud account, no extra perms |

> **Version vs. version code.** `expo.version` is the human-readable name
> ("1.0.0"). Google Play also requires an integer `versionCode` that must
> strictly increase with every upload. Because `eas.json` has
> `cli.appVersionSource: "remote"` and the `production` profile uses
> `autoIncrement: true`, EAS owns the `versionCode` for you — it
> auto-increments on every cloud build and stores the counter in your EAS
> account. **You don't need to edit `android.versionCode` in `app.json`.**

> **First-release version.** The Play Store rejects releases with version
> `0.x.x` as "alpha-looking" only in spirit, not by policy — but it's
> strongly recommended to ship `1.0.0` for the first production release.
> Bump [app.json](app.json) before the first `eas build --profile production`.

Then verify the EAS profile in [eas.json](eas.json):

```json
"production": {
  "channel": "production",
  "autoIncrement": true
}
```

(No `distribution: "internal"` here — that flag is what makes `preview`
emit an `.apk` for sideloading. Without it, `production` emits a Play-ready
`.aab`.)

## Step 2 — Create the app in Play Console

1. Go to https://play.google.com/console → **Create app**.
2. Fill in:
    - **App name:** `Spendium`
    - **Default language:** English (United States) — `en-US`
    - **App or game:** App
    - **Free or paid:** Free
    - **Declarations:** check both (Play policies + US export laws)
3. Click **Create app**.
4. Note your app's **package name** is now bound: it must match
   `expo.android.package` (`com.vshpynta.spendium`) on every upload. You
   cannot change either side later — a different package name = a different
   Play Store listing.

You'll now land on the app's **Dashboard**. The left rail has a "Set up
your app" checklist. We'll work through it in the next steps.

## Step 3 — Build a Play-Store-ready AAB

The mobile `README.md` already documents the **one-time EAS setup** (free
account, `npx eas login`, `npx eas project:init`, keystore generation on
first build, OAuth client IDs). If you haven't done it yet, follow the
[One-time setup](README.md#one-time-setup) section there first — Play
deployment reuses the exact same keystore and account as a `preview`
sideload build, so the setup is shared.

Two build paths produce a Play-ready `.aab`. They are mutually exclusive
— pick one and skip the other.

| Option   | Where the build runs | Toolchain you need               | Keystore managed by                         | Best for                                                                                  |
|----------|----------------------|----------------------------------|---------------------------------------------|-------------------------------------------------------------------------------------------|
| **A** — EAS Cloud | Expo's servers       | `eas-cli` only                  | Expo (auto-generated, downloadable)         | First release; CI; any machine without an Android SDK                                     |
| **B** — EAS Local | Your machine         | `eas-cli` + Android SDK + JDK 17 | Expo (same keystore as Option A)            | Faster iteration when you have Android Studio installed; avoids EAS Cloud queue / limits  |

Both produce the same artifact — a signed `.aab` — and feed into the same
[Step 4](#step-4--first-upload-manual-to-internal-testing) upload and
[Step 5](#step-5--wire-up-eas-submit-for-future-releases) automation. The
`production` profile differs from `preview` (sideloading) only in two
ways: output format (`.aab` vs `.apk`) and `autoIncrement: true` on
`versionCode`.

### Option A — EAS Cloud Build (recommended for first release)

Nothing to install locally beyond `eas-cli`. From
`expenses-tracker-mobile/`:

```powershell
npx eas whoami                                                # confirm you're logged in
npx eas build --platform android --profile production
```

**First-run prompts** (only on your very first `production` build for
this package):

1. *"Generate a new Android Keystore?"* → **Yes**. EAS creates the
   upload key, stores it server-side under your Expo account, and reuses
   it on every subsequent build (cloud or local).
2. *"What would you like your Android application id to be?"* — if it
   asks, accept the default (`com.vshpynta.spendium` from
   [app.json](app.json)). It should not ask if `expo.android.package` is
   already set.

**While it runs** the CLI prints a build-dashboard URL like
`https://expo.dev/accounts/<you>/projects/expenses-tracker-mobile/builds/<id>`.
Open it to watch logs in real time; the build is queued for ~30–60 s on
the free tier, then runs ~10–15 min.

**When it finishes:**

- The CLI prints a download URL
  (`https://expo.dev/artifacts/eas/<hash>.aab`). The same link is on
  the build dashboard under *Artifacts*.
- Download the `.aab` (browser, or
  `Invoke-WebRequest -Uri <url> -OutFile spendium-1.aab`). Save it
  somewhere you'll find again — you'll need it for the manual upload
  in [Step 4](#step-4--first-upload-manual-to-internal-testing).
- Verify the size is plausible (Spendium AABs land around 30–60 MB).
  Use `(Get-Item .\spendium-1.aab).Length / 1MB` to print it. A
  multi-hundred-MB or sub-1 MB file means something went wrong.
- Optional: `npx eas build:list --platform android --limit 1` re-prints
  the metadata (versionName, versionCode, status) if you lost the
  terminal output.

> **Free-tier limits.** Expo's free plan caps Android cloud builds at
> 30/month with no priority queue. If you hit the cap or queue is slow,
> switch to Option B for the rest of the month — same keystore, same
> versionCode counter, runs on your machine.

### Option B — EAS Local Build (`--local` flag)

Same command as Option A, `--local` flag added. EAS still owns the
keystore and the `versionCode` counter (so you must still be `eas login`'d
— it pulls credentials down on the first run), but the Gradle build
itself runs on your machine using the Android SDK + JDK 17 that Android
Studio installs.

**Prereqs** (verify *before* running the build). Android Studio installs
all of these automatically; these commands confirm they're on `PATH`:

```powershell
java -version                              # must report 17.x; not 21, not 11
echo $env:ANDROID_HOME                     # must print the SDK path (e.g. C:\Users\<you>\AppData\Local\Android\Sdk)
Test-Path "$env:ANDROID_HOME\platforms\android-35"      # must be True
Test-Path "$env:ANDROID_HOME\build-tools\35.0.0"        # must be True
npx eas whoami                             # must print your Expo username, not 'Not logged in'
```

If any check fails: open Android Studio → **More Actions → SDK Manager**
→ *SDK Platforms* tab and tick **Android 15 (API 35)**; *SDK Tools* tab
and tick **Android SDK Build-Tools 35.0.0**. For JDK 17, point
`JAVA_HOME` at Android Studio's bundled JDK
(`%LOCALAPPDATA%\Programs\Android Studio\jbr` on Windows) or install
[Eclipse Temurin 17](https://adoptium.net/temurin/releases/?version=17).

**Run the build** from `expenses-tracker-mobile/`:

```powershell
npx eas build --platform android --profile production --local
```

**Artifact location.** The signed `.aab` is written to the working
directory as `expenses-tracker-mobile/build-<epoch>.aab` (e.g.
`build-1717123456789.aab`). The epoch suffix is unique per build, so you
can tell releases apart by date — no need to rename it.

**Timing.** The first `--local` build is slow (~20–30 min on Windows)
because Gradle downloads ~2 GB of dependencies, the Android NDK, and
the React Native build chain, and writes them into
`%USERPROFILE%\.gradle`. Subsequent builds reuse the cache and finish
in 4–8 min. Free ≥10 GB of disk before the first run.

> **Keystore backup.** The first `eas build` for this `package` generates
> the **upload key** and stores it in your Expo account. For Play Store
> deployment the stakes are higher than for sideloading: lose the keystore
> and Google won't let you ship updates. Run
> `npx eas credentials -p android` → **Download keystore** to back up the
> `.jks` and its passwords in a password manager.

## Step 4 — First upload (manual) to Internal Testing

The very first upload **must** go through the Play Console UI because the
upload key fingerprint has to be registered with Play App Signing before
`eas submit` can automate it.

1. Play Console → your app → **Test and release → Internal testing**
   → **Create new release**.
2. Section **App integrity** → on the first release Google will prompt
   you to **enroll in Play App Signing** — accept (it's the only option
   for new apps).
3. Section **App bundles** → upload the `.aab` from Step 3 (the file you
   downloaded for Option A, or `build-<epoch>.aab` from the
   `expenses-tracker-mobile/` directory for Option B).
4. Section **Release details**:
    - **Release name:** auto-filled from the version (e.g. `1.0.0 (1)`)
    - **Release notes (en-US):** something like
      ```
      Initial release. Private expense tracker with Google Drive and
      OneDrive sync. Works fully offline.
      ```
5. Click **Next** → review warnings → **Save**, then **Review release** →
   **Start rollout to Internal testing**.
6. While still on Internal testing, **Testers** tab → create an email
   list and add yourself + any other testers. Share the **opt-in URL** —
   testers must open it on the same Google account that's on their device.

This isn't visible to the public yet. Now skip ahead to **Step 6** to
finish the store listing — then loop through Steps 7–9 to clear the
required Play Console questionnaires before promoting to Production.
Step 5 (automating future uploads) is **safe to defer**: nothing here
gates it, and you'll only need it for the *second* release onwards.

## Step 5 — Wire up `eas submit` for future releases

Once the first upload is on Internal testing, automate subsequent
uploads. This needs a Google Cloud **service account** with Play Console
permissions.

1. **Create the service account** (one-time):
    - In Google Cloud Console, create a project (or use an existing one
      linked to your Play developer account).
    - Enable the **Google Play Android Developer API**.
    - IAM → **Service accounts** → create one (e.g.
      `eas-submit@<your-project>.iam.gserviceaccount.com`).
    - Create a **JSON key** for that account and download it. Treat it
      like a password.
2. **Grant Play Console access**:
    - Play Console → **Users and permissions** → **Invite new users** →
      enter the service-account email → grant **Admin (all permissions)**
      for this app only (least privilege; finer grants also work but Admin
      is the easiest combo for `eas submit`).
3. **Wire it into `eas.json`**:

    ```json
    "submit": {
      "production": {
        "android": {
          "serviceAccountKeyPath": "./play-store-service-account.json",
          "track": "internal"
        }
      }
    }
    ```

4. Save the JSON key at the path above (mobile root) and **add it to
   `.gitignore`** — it must never be committed.

5. Use it:

    ```powershell
    # Build and submit in one shot (Options A & B from Step 3):
    npx eas build --platform android --profile production --auto-submit

    # Or submit an existing artifact from EAS Cloud:
    npx eas submit --platform android --latest

    # Or submit a locally-built .aab (Option B):
    npx eas submit --platform android --path ./build-*.aab
    ```

After this, every release reduces to: bump `expo.version` →
`eas build --auto-submit` (or local build + `eas submit --path`) → write
release notes in Play Console → roll out.

## Step 7 — Complete the Data safety form

Play Console → **App content → Data safety** → **Start**.

Spendium's posture is unusual — most fields are **"No data collected"**.
Here's the verbatim answer for each section:

| Section                                         | Answer                                                                                                                              |
|-------------------------------------------------|-------------------------------------------------------------------------------------------------------------------------------------|
| Does your app collect or share any user data?   | **Yes** — because OAuth tokens and Drive/OneDrive file IDs technically count as "Personal info → User IDs" even though stored only on-device. |
| Is all user data collected by your app encrypted in transit? | **Yes** — Google Drive / Microsoft Graph APIs are HTTPS-only.                                                          |
| Do you provide a way for users to request that their data be deleted? | **Yes** — Settings → Reset / sign out + uninstall removes local DB; the user controls their own Drive/OneDrive file. |
| **Personal info → User IDs**                    | Collected: **Yes**. Shared: **No**. Purpose: **App functionality** (auth). Optional: **Yes** (cloud sync is opt-in — users can use the app fully offline and no OAuth tokens are ever stored). Ephemeral: **No** (tokens persisted in Keychain/Keystore). |
| **Financial info → User payment info / Purchase history / Credit score / Other financial info** | **NOT collected** — expense entries live only on the device and in the user's own cloud folder; the app never sees them. |
| **Files and docs**                              | **NOT collected** — Drive file is created in the user's `appDataFolder` (Spendium can only see files it created). |
| **App activity / App info and performance / Device or other identifiers / Health / Location / Web browsing / Contacts / Calendar / Messages / Photos & videos / Audio / Personal info (other)** | **NOT collected**.                                              |

After saving, the form must be in state **Complete** before you can
promote a release to production.

## Step 8 — Content rating, target audience, privacy policy

These are gated by Play Console — your first production rollout will
refuse to proceed until all three are green.

| Item                  | Where                                              | Notes                                                                                                                              |
|-----------------------|----------------------------------------------------|------------------------------------------------------------------------------------------------------------------------------------|
| **Content rating**    | App content → Content ratings → **Start questionnaire** | Spendium answers "No" to every category (no violence/gambling/etc.). Final IARC rating will be **Everyone**.                  |
| **Target audience**   | App content → Target audience and content          | Select 18+ (Spendium has no kid-friendly mode; targeting 13+ enables stricter ad/data rules you don't need).                       |
| **Privacy policy**    | App content → Privacy policy                       | Paste the same HTTPS URL you used in the listing. Must be reachable; Google's crawler verifies it.                                |
| **Ads**               | App content → Ads                                  | **No, my app does not contain ads.**                                                                                              |
| **News app**          | App content → News apps                            | **No.**                                                                                                                            |
| **Government app**    | App content → Government apps                      | **No.**                                                                                                                            |
| **Financial features**| App content → Financial features                   | **No** — Spendium isn't a payments/loans/investments app.                                                                          |
| **Data deletion**     | App content → Data deletion                        | Explain: "Uninstall the app to remove all on-device data. Sign out via Settings → Cloud sync to revoke cloud access. Delete the `appDataFolder` file via Drive UI to remove the cloud copy." |

## Step 9 — Promote Internal → Closed → Open → Production

Google strongly recommends staging through testing tracks before going
public. Each track uses the same `.aab`; you only choose **which track**
the release lands on.

| Track                | Audience                                     | When to use                                                                                                                       |
|----------------------|----------------------------------------------|-----------------------------------------------------------------------------------------------------------------------------------|
| **Internal testing** | Up to 100 testers you list by email          | Smoke test the build on real devices via the Play Store install flow (catches things that local sideloading misses, e.g. App Signing). |
| **Closed testing**   | Larger tester groups (e.g. 100s of users)    | Required for first-time publishers: Google now demands ~20 testers run a Closed track for ~14 days before allowing Production.    |
| **Open testing**     | Anyone with the opt-in URL                   | Optional public beta.                                                                                                              |
| **Production**       | Everyone in the targeted countries           | Final release.                                                                                                                     |

Promotion is one click: **Internal testing → Promote release → Production**
(or any intermediate track). The `.aab` is reused; only the track changes.

> **First-app reality check.** As of late 2024, new Google Play developer
> accounts are required to run a **Closed testing track with at least 20
> opted-in testers for 14 days** before being eligible to publish to
> Production. Plan for this gating — share the opt-in URL with at least
> 20 people the day you create the Closed track.

## Step 10 — Shipping subsequent updates

```powershell
# 1. Bump the human-readable version in app.json (e.g. "1.0.0" -> "1.1.0").
#    versionCode auto-increments via EAS — don't touch it.

# 2. Build + submit. Pick the path you used in Step 3:

# Option A (EAS Cloud):
npx eas build --platform android --profile production --auto-submit

# Option B (EAS Local):
npx eas build --platform android --profile production --local
npx eas submit --platform android --path ./build-*.aab

# 3. In Play Console -> Internal testing, write release notes (per locale)
#    and start rollout to Internal testing.

# 4. Smoke-test on a tester device. When happy:
#    Internal testing -> Promote release -> Production -> Start staged rollout.
```

Staged rollout starts at 1–10 % of users and you bump it up over days.
If a crash spike shows in Play Console → Quality → Android vitals,
**Halt rollout** stops new installs immediately (existing installs keep
the bad build until you ship a fix).

## Optional — EAS Update (OTA JS-only patches)

For JavaScript-only fixes (a typo, a logic bug in `src/`, no native
changes), you can push an over-the-air update without going through the
Play Store review:

```powershell
npx eas update --channel production --message "fix: typo in settings"
```

This requires `expo-updates` to be installed and configured. **Native
changes** (new dependency that ships native code, `app.json` plugin
tweaks, icon updates, permission changes) still require a fresh
`eas build` + Play Store release — EAS Update can only swap the JS
bundle.

Configure once via:

```powershell
npx expo install expo-updates
npx eas update:configure
```

…then add `runtimeVersion` to [app.json](app.json) so a JS bundle is only
served to compatible native builds.

## Troubleshooting

| Symptom                                                              | Cause                                                                                          | Fix                                                                                                                                                                                                                                                                       |
|----------------------------------------------------------------------|------------------------------------------------------------------------------------------------|-----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| `Version code X has already been used`                               | EAS's auto-increment got out of sync with what Play already has (e.g. you uploaded manually).  | Run `npx eas build:version:set` (interactive) to align the EAS counter with the highest value Play has seen. The flag form `--platform android --value 42` skips the prompts. Alternatively, switch `cli.appVersionSource` to `"local"` in [eas.json](eas.json) and set `expo.android.versionCode` in [app.json](app.json) directly — then EAS reads from there instead of its remote counter. |
| `Your app is using a non-compliant version of the Play Core library` | A transitive RN dep still imports the legacy Play Core API.                                    | Bump to Expo SDK 55+ (already on it). If the error persists, run `npx expo-doctor` and follow its upgrade hints.                                                                                                                                                       |
| `App not installed: package conflicts with an existing package`      | A sideloaded `preview` APK on the test phone uses the same package id but a different signing key. | Uninstall the sideloaded copy first: `adb uninstall com.vshpynta.spendium`. Play-installed and dev-signed builds can't coexist on one device.                                                                                                                          |
| Play Console: "Privacy policy URL is required"                       | The URL field is empty or returns non-200 to Google's crawler.                                 | Make sure the URL is HTTPS and publicly reachable (no Cloudflare bot-blocker, no auth wall, no robots-disallow).                                                                                                                                                       |
| Play Console: "Your app cannot be promoted to production"            | First-publisher 20-tester / 14-day Closed-track requirement not met.                           | Open Closed testing → ensure ≥ 20 opted-in testers → wait the full 14 days. There is no opt-out.                                                                                                                                                                       |
| `eas submit` fails with `Permission denied`                          | The service account doesn't have Play Console access yet, or its permissions weren't saved.    | Play Console → Users and permissions → confirm the service-account email is listed and "App access" includes your app. Permission changes can take a few minutes to propagate.                                                                                         |
| OAuth (Google Drive / OneDrive) breaks only in the Play-Store build  | Play App Signing rewrote the signing certificate, so the redirect handler `com.vshpynta.spendium:/oauth2redirect` is now signed with a different SHA-1/SHA-256 than your registered OAuth client. | In Google Cloud Console → APIs & Services → Credentials → your Android OAuth client, add Play's **app signing key** SHA-1 (visible in Play Console → Setup → App integrity → App signing key certificate).
